Once your toolchain is in place — see [Prerequisites](/docs/getting-started/prerequisites) —
you can clone the repo and run a single setup command.

<Callout title="Windows users" type="warn">
  Run `bun run setup:windows` *before* `setup:all` and restart your terminal.
  The Windows bootstrap checks for the MSVC/Tauri build tools and prints install
  guidance if they are missing. See [Platform
  setup](/docs/getting-started/platform-setup#windows).
</Callout>

## Quick setup

<Steps>
  <Step>
### Clone the repository

```bash
git clone https://github.com/Bongisto/SabbathCue.git
cd SabbathCue
```

  </Step>
  <Step>
### Install JavaScript dependencies

```bash
bun install
```

This pulls in the React frontend, the build orchestrator, and the data-prep
scripts.

  </Step>
  <Step>
### Run the unified setup pipeline

```bash
bun run setup:all
```

`setup:all` is **idempotent** — re-running it skips phases whose
output artifacts already exist, so it's safe to invoke after
`git pull` or when something changed in `data/`. Pass `--force` to
re-run every phase. The orchestrator (`data/prepare-embeddings.ts`)
runs eight phases in this exact order:

1. **Python environment** — creates `.venv` and installs the pip
   dependencies (`optimum-onnx[onnxruntime]`,
   `sentence-transformers`, `accelerate`, `tokenizers`, `numpy`,
   `torch`, `meaningless`).
2. **Bible source data** — downloads a single bundled archive that
   contains all ten translations (KJV, NIV, ESV, NASB, NKJV, NLT,
   AMP, SpaRV, FreJND, PorBLivre) plus the cross-reference dataset
   from openbible.info.
3. **SQLite database** — assembles `data/rhema.db` with FTS5
   virtual tables and cross-reference tables.
4. **EGW import** - imports EGW books into `data/rhema.db`.
5. **ONNX model** - downloads the embedding model, exports it with
   the sentence-transformers ONNX backend, and quantizes it to INT8.
6. **Verse export** - emits KJV/NKJV/NLT verses to JSON for the
   embedding step.
7. **Embedding precomputation** - runs the embeddings (auto-selects
   GPU via PyTorch if available; falls back to ONNX Runtime on CPU).
8. **Local STT assets** - prepares the local Vosk model files used by
   the default offline provider (`bun run download:vosk`).

  </Step>
  <Step>
### Run in development

```bash
bun run tauri dev
```

The Tauri shell opens with hot-reload for the React frontend at port 3000
and live-reloads the Rust backend on file changes.

  </Step>
</Steps>

## Build for production

When you're ready to ship a native installer:

```bash
bun run tauri build
```

The output lands in `src-tauri/target/release/bundle/` — `.dmg` on macOS,
`.msi` and `.exe` on Windows, `.AppImage` and `.deb` on Linux.

## NDI SDK (optional)

The NDI SDK isn't bundled because of NDI's redistribution policy. If you
want broadcast output, fetch it once:

```bash
bun run download:ndi-sdk
```

This drops the platform-specific NDI 6 headers and shared libraries under
`sdk/ndi/`, where the `rhema-broadcast` crate finds them via dynamic
loading.

## Running individual phases

Sometimes you want to re-run a single phase without nuking the whole
pipeline. Each phase has its own script:

<Tabs items={["Bun scripts", "Direct Python"]}>
  <Tab value="Bun scripts">

```bash
bun run download:bible-data        # Bundled translations + cross-refs
bun run build:bible                # Build SQLite database
bun run download:model             # Download & export ONNX model
bun run export:verses              # Export verses to JSON
bun run precompute:embeddings      # Rust ONNX binary (recommended)
bun run precompute:embeddings-onnx # Python ONNX Runtime path
bun run precompute:embeddings-py   # Python sentence-transformers (GPU)
bun run download:vosk              # Default local Vosk model
bun run build:vosk-sidecar         # Self-contained Vosk worker for releases
```

  </Tab>
  <Tab value="Direct Python">

```bash
python3 data/precompute-embeddings.py      # GPU sentence-transformers
python3 data/precompute-embeddings-onnx.py # CPU ONNX Runtime fallback
```

  </Tab>
</Tabs>

See the [scripts reference](/docs/reference/scripts) for the full list and
when to reach for each one.
