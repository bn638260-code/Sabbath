Live operators rarely have a free hand on a keyboard. SabbathCue exposes its
controls over **OSC** and an **HTTP API** so you can wire them into
hardware controllers, automation scripts, and stage-management
software. Both protocols can run simultaneously and share the same
command set.

<Callout title="In-depth guide" type="info">
This page is a high-level overview. The full protocol reference,
authentication notes, and example payloads live in the
[remote control guide](https://github.com/Bongisto/SabbathCue/blob/main/documentation/remote-control.md)
in the repository.
</Callout>

## Protocols at a glance

| Protocol | Default port | Transport | Best for |
| --- | --- | --- | --- |
| OSC | `8000` | UDP | Stream Deck (via Companion), TouchOSC, QLab, hardware |
| HTTP | `8080` | TCP | REST clients, dashboards, scripting, status polling |

Both default to binding `127.0.0.1`, so remote control is local to the app
machine unless you intentionally front it with trusted network infrastructure.
The current Settings UI exposes listener ports, not a LAN bind-host control.

## What you can do remotely

| Action | OSC address | HTTP body (POST `/api/v1/control`) |
| --- | --- | --- |
| Next verse in queue | `/sabbathcue/next` | `{ "command": "next" }` |
| Previous verse | `/sabbathcue/prev` | `{ "command": "prev" }` |
| Show / hide overlay | `/sabbathcue/show`, `/sabbathcue/hide` | `{ "command": "show" }` |
| Switch theme | `/sabbathcue/theme "Classic Dark"` | `{ "command": "theme", "value": "Classic Dark" }` |
| Set opacity | `/sabbathcue/opacity 0.75` | `{ "command": "opacity", "value": 0.75 }` |
| Set confidence threshold | `/sabbathcue/confidence 0.8` | `{ "command": "confidence", "value": 0.8 }` |
| Toggle on-air | `/sabbathcue/on_air true` | `{ "command": "on_air", "value": true }` |

All eight commands work identically across both protocols, so any
controller you already own (or build) talks the same dialect.

<Callout title="Opacity is a stub today" type="warn">
The `opacity` command is wired through the dispatcher but currently a
placeholder — it'll take effect when the broadcast store gains opacity
support. The other seven commands are fully live.
</Callout>

## OSC

OSC runs on UDP port `8000`. Configure the listening port in
Settings → Remote, and send messages from your favorite controller:

- **TouchOSC** templates with verse-show/hide pads, opacity sliders,
  and theme buttons drop right onto a tablet.
- **Stream Deck** + Bitfocus Companion's Generic OSC module maps each
  pad to one of the addresses above.
- **A simple test from the command line**:

  ```bash
  oscsend localhost 8000 /sabbathcue/next
  ```

The Settings panel shows a **live command log** with timestamps so you
can debug controller setups without leaving the app.

## HTTP API

For scripts and web automations, the HTTP API exposes the same
controls under `/api/v1/`:

| Endpoint | Method | Purpose |
| --- | --- | --- |
| `/api/v1/health` | GET | Liveness check — returns service name and version |
| `/api/v1/status` | GET | Current state snapshot — on-air, active theme, live verse, queue length, confidence threshold |
| `/api/v1/control` | POST | Dispatch a command (table above) |

```bash
curl -X POST http://localhost:8080/api/v1/control \
  -H 'Content-Type: application/json' \
  -d '{"command":"theme","value":"Classic Dark"}'
```

The status endpoint returns an authoritative view from a Rust-side
state cache that the frontend refreshes once a second:

```bash
curl http://localhost:8080/api/v1/status
```

```json
{
  "on_air": true,
  "active_theme": "Classic Dark",
  "live_verse": "John 3:16",
  "queue_length": 12,
  "confidence_threshold": 0.75
}
```

## Persistence and shutdown

OSC and HTTP listeners can be started and stopped without restarting
the app. Port-bind failures surface as visible errors in the Settings
panel rather than as silent listener crashes, and both listeners shut
down cleanly on app exit (no zombie threads, no orphaned ports).

## Security

By default, both protocols bind to `127.0.0.1`. OSC is local-only and
unauthenticated. HTTP requires a bearer token for status and control calls, with
`/api/v1/health` left open for liveness checks. Before exposing SabbathCue
beyond the app machine, front it with a reverse proxy or private overlay network
that adds authentication, TLS, and access control. The repo's
[remote control guide](https://github.com/Bongisto/SabbathCue/blob/main/documentation/remote-control.md#security-considerations)
covers the threat model in more depth.
